# 4090 服务器连接故障排查记录

日期：2026-05-20

## 目标服务器

- 原登录方式：`ssh root@120.26.176.56 -p 10009`
- 项目目录：`/workspace/thymic_project`

## 当前检测结果

- `120.26.176.56` 可以 ping 通，说明公网 IP 本身在线。
- `120.26.176.56:10009` TCP 不通，返回连接拒绝。
- `120.26.176.56:22` TCP 可通，但不是我们平时使用的免密训练入口；临时 known_hosts 测试后返回 `Permission denied (publickey,password)`。
- 本机 SSH 配置里的 SeeTaCloud 历史端口已批量尝试，未发现可免密进入并访问 `/workspace/thymic_project` 的入口。

## 判断

这次不是单纯 `/workspace` 挂载坏导致的训练目录不可用。

昨天的问题是：SSH 能进入服务器，但 `/workspace` 报 `Transport endpoint is not connected`，属于对象存储/FUSE 挂载异常。

今天的问题更靠前：`10009` 登录端口本身已经连不上。常见原因是：

- 实例/容器没有正常运行。
- 平台侧 SSH 端口转发失效。
- 服务器重启后 SSH 登录端口变化。
- 平台回收或重建了实例，原 `10009` 已不是当前入口。

## 需要从平台侧处理的步骤

1. 进入 SeeTaCloud / 云平台控制台，确认这台 4090 实例是否处于运行状态。
2. 如果实例显示运行但 SSH 不通，优先点一次重启实例或重启 SSH/端口转发。
3. 重新复制平台生成的最新 SSH 登录命令，不要默认继续使用 `120.26.176.56:10009`。
4. 如果平台提供 Web Terminal，先用 Web Terminal 进入后执行：

```bash
hostname
uptime
nvidia-smi
df -h / /workspace
ls -ld /workspace /workspace/thymic_project
findmnt /workspace || true
ps -ef | grep -E 'ossfs|fuse|ssh' | grep -v grep || true
```

## 服务器恢复登录后的修复路线

如果能重新 SSH 进入，但 `/workspace` 仍然报 `Transport endpoint is not connected`：

1. 先确认没有训练进程仍在使用 `/workspace`。
2. 对坏掉的 FUSE 挂载做 lazy unmount。
3. 再用平台原始挂载方式重新挂载对象存储。
4. 挂载恢复后检查 `/workspace/thymic_project`、conda 环境和 GPU。
5. 再启动 Task6 curriculum 训练脚本。

注意：现在还不能直接执行第 2 步，因为我们还没有进入服务器，也不知道平台当前实际挂载命令。
